  - **Composition with `--polish` is bidirectional.** `--polish "<reason>" --plan` writes a polish-pass plan (revision mode field equals `polish` in the plan header; verbatim operator reason recorded as a header subsection); `--apply` reads the plan's revision_mode + operator reason and threads them through to the produced version dir's `metadata.revision_mode = "polish_plan_then_apply"` + `metadata.revise_force_reason = <verbatim>`. The operator does NOT re-pass `--polish` on the `--apply` invocation — the plan IS the audit trail. The `polish_plan_compose/` fixture exercises this composition end-to-end.
  - **`--plan` and `--apply` are mutually exclusive.** Passing both in the same invocation is rejected with a clear error. The two-phase contract is sequential: write the plan, optionally edit it, then apply it. A "fire-and-forget" mode that combines both (write a plan and immediately apply it without operator review) would defeat the purpose of the preview gate.
  - **Three accepted rejection shapes** on `plan.md` (operator edits in place between `--plan` and `--apply`): (1) same-line `<!-- declined: <reason> -->` comment appended to the table row; (2) row deletion (treated as `Resolution: declined — removed from plan` at apply time); (3) `Priority: declined` cell value + `[declined: <reason>]` bracketed addition to the `Summary` cell. Each shape becomes a `Resolution: declined — <reason>` row in `<thread>.{N+1}/changelog.md`. The reason flows verbatim — `--apply` MUST NOT paraphrase or shorten. The shape variety exists because operator editing styles diverge (some operators prefer comments, some prefer cell edits, some prefer row deletion); the parser is small enough to handle all three.
  - **Plan staleness contract (5 cases)**: `--apply` refuses the plan when (1) no matching plan exists, (2) source review `verdict.md` mtime is later than `plan.md`'s (verdict was re-run since plan write), (3) a new critic sibling exists that wasn't recorded in `_progress.json.metadata.critic_siblings_at_plan_time`, (4) `plan.md` mtime is older than `plan_max_age_days` (default 7, configurable via `<thread>/.anvil.json`), or (5) `<thread>.{N+1}/memo.md` already exists. Each rejection points at remediation (typically: re-run `--plan` to refresh). The staleness contract is what makes the two-phase flow safe against the most common drift sources (the operator re-running review or adding a new critic mid-flow).
